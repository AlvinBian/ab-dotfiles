# Incident 處理流程

## Step 1 — 定級與影響評估

立即確認：

| 問題 | 回答 |
|------|------|
| 哪個服務/功能受影響？ | |
| 影響範圍（全部/部分用戶）？ | |
| 開始時間？ | |
| 有無錯誤訊息/截圖？ | |

**定級：**
- 🔴 **P0** — 核心功能完全不可用，影響所有用戶
- 🟡 **P1** — 重要功能降級，影響部分用戶
- 🟢 **P2** — 非核心功能異常，有替代方案

## Step 2 — 快速診斷

依序檢查（不要同時做多件事）：

```bash
# 1. 確認服務狀態
curl -I https://{service}/health

# 2. 看最近部署
git log --oneline -10
# 或
{平台} releases list --limit 5

# 3. 看錯誤日誌（最近 30 分鐘）
# 依平台調整：kubectl logs / pm2 logs / heroku logs

# 4. 確認基礎設施
# DB 連線？Cache 連線？外部 API？
```

**假設驅動**：列出 3 個最可能原因，從最可能開始排查。

## Step 3 — 止血（先恢復，再找原因）

按優先順序嘗試：

1. **回滾** — 如果最近有部署，先回滾觀察
2. **繞過** — 關閉有問題的功能開關、切換備援
3. **擴容** — 如果是負載問題，先擴 instance
4. **重啟** — 最後手段，確認不會造成資料問題

> ⚠️ 止血操作執行前先在頻道同步，每個操作都記錄時間。

## Step 4 — 溝通模板

**初始通報**（發現後 5 分鐘內）：
```
🔴 [P{級別}] {服務} 異常

影響：{描述影響範圍}
開始：{時間}
狀態：調查中

處理人：@{你}
下次更新：15 分鐘後
```

**進度更新**（每 15-30 分鐘）：
```
📍 更新 {時間}

已確認：{發現什麼}
正在做：{當前行動}
預計：{預計恢復時間或下一步}
```

**恢復通報**：
```
✅ [已恢復] {服務} 恢復正常

恢復時間：{時間}
總耗時：{X} 分鐘
根本原因：{簡述}
後續：{預防措施，詳見復盤}
```

## Step 5 — 復盤報告

事後 24-48 小時內產出：

```markdown
# 事故復盤：{標題}

## 摘要
- 時間線：{開始} → {發現} → {止血} → {恢復}
- 影響：{用戶數/功能/時長}
- 根本原因：{一句話}

## 時間線
| 時間 | 事件 | 執行人 |
|------|------|--------|
| HH:MM | 收到告警 | |
| HH:MM | 確認影響範圍 | |
| HH:MM | 開始回滾 | |
| HH:MM | 服務恢復 | |

## 根本原因分析（5 Why）
1. 為什麼發生？{直接原因}
2. 為什麼？{上一層原因}
3. 為什麼？{再上一層}
...

## 行動項目
| 項目 | 負責人 | 截止日 |
|------|--------|--------|
| {預防措施} | | |
| {監控改善} | | |
| {流程改善} | | |

## 經驗教訓
- 做得好的：{保留}
- 可改進的：{改進點}
```
