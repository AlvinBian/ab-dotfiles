# ADR — 架構決策記錄

## 何時用 ADR

| 情境 | 要不要寫 ADR |
|------|------------|
| 選擇技術框架 / 資料庫 / 第三方服務 | ✅ |
| 選擇架構模式（微服務 vs 單體、REST vs GraphQL）| ✅ |
| 棄用或替換既有方案 | ✅ |
| 有爭議、需要對齊共識的決策 | ✅ |
| 未來可能回頭質疑這個決定的情境 | ✅ |
| 日常 Bug 修復 | ❌ |
| 無爭議的明確做法 | ❌ |

## 使用時機與流程

### 發起 ADR 的流程

1. **提出** — 任何人都可以提出，狀態設為 `Proposed`
2. **討論** — 在 PR 或文件中收集意見（通常 3-5 天）
3. **決定** — 決策者確認，狀態改為 `Accepted`
4. **存檔** — 放入 `docs/adr/ADR-{序號}-{slug}.md`

### 目錄命名規範

```
docs/
└── adr/
    ├── ADR-0001-use-postgresql.md
    ├── ADR-0002-microservices-vs-monolith.md
    └── ADR-0003-graphql-api-design.md
```

### 狀態說明

| 狀態 | 說明 |
|------|------|
| `Proposed` | 草稿，等待討論和決策 |
| `Accepted` | 已決策，生效中 |
| `Deprecated` | 不再適用，但決策本身沒被取代 |
| `Superseded by ADR-{n}` | 被新 ADR 取代 |

## 快速模板（精簡版）

適合小決策或時間緊迫時：

```markdown
# ADR-{序號}：{標題}

**狀態**：Accepted | **日期**：{date}

**問題**：{一句話說問題}

**選項**：A={方案A}，B={方案B}

**決策**：選 {A/B}，因為 {核心理由一句話}

**後果**：{主要影響}，**風險**：{主要風險}
```

## 提示

執行此 command 時，請先詢問用戶：

1. **決策主題是什麼？**（例：「要不要用 Redis 做 Session 快取」）
2. **已知的選項有哪些？**（或讓我協助列出）
3. **最重要的評估維度是什麼？**（效能/成本/團隊熟悉度/...）
4. **ADR 序號**（看現有 docs/adr/ 最後一個 + 1）

然後依據回答生成完整 ADR。
